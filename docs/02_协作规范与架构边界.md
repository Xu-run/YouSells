# YouSells 协作规范与架构边界

日期：`2026-05-19`

## 1. 开发流程

所有任务都按下面顺序推进：

1. 明确需求
2. 确认开发范围
3. 记录开发计划或日志
4. 编写模块代码
5. 立刻补自动化测试
6. 运行验证
7. 涉及页面或接口时做前后端联调

不要跳过需求和范围确认直接写代码。一个 PR 只解决一个任务包，不做无关顺手重构。

## 3. 后端架构边界

后端采用单体模块化结构：

```text
yousells-server/src/main/java/com/yousells/
├── common/
└── modules/
    ├── auth/
    ├── customer/
    ├── followup/
    ├── task/
    ├── report/
    ├── script/
    └── dashboard/
```

模块职责：

- `common`：通用配置、常量、异常、统一响应、安全上下文
- `auth`：登录、当前用户、登出、JWT 生成
- `customer`：客户、客户标签、客户详情
- `followup`：跟进记录
- `task`：公共安排
- `report`：日报 / 周报
- `script`：话术库
- `dashboard`：首页聚合，只做聚合，不承载业务持久化

## 4. 后端分层规范

每个业务模块按下面结构落文件：

```text
module/
├── controller/
├── service/
├── service/impl/
├── dto/
├── vo/
├── entity/
├── mapper/
└── convert/
```

分层职责：

- `controller`：只处理 HTTP 入参、校验、返回，不写业务逻辑
- `service`：承载业务编排和模块对外能力
- `dto`：请求参数
- `vo`：返回结构
- `entity`：数据库实体
- `mapper`：MyBatis-Plus 数据访问
- `convert`：对象转换

## 5. 跨模块调用规则

允许：

- `controller -> service`
- `service -> 本模块 mapper`
- `dashboard.service -> customer/followup/task service`
- `业务模块 -> common`

不允许：

- 直接跨模块调用 mapper
- controller 互相调用
- 在 controller 写业务逻辑
- 在 dashboard 里写客户、跟进、任务的持久化逻辑
- 为了图快绕过 service 接口

## 7. 测试要求

后端：

- service 逻辑必须有单元测试或聚合测试
- controller 行为建议有集成测试
- 新增 mapper 后必须验证基础 CRUD 或关键查询
- 鉴权、权限、异常路径必须覆盖

前端：

- 页面开发完成后必须本地打开页面自测
- 表单提交、列表刷新、跳转、空状态必须检查
- 调用后端接口的页面必须完成前后端联调

## 8. PR 要求

合并前至少满足：

- 只改自己负责范围
- 测试通过
- 构建通过
- 接口字段变化已更新文档
- 页面变化已完成浏览器联调
- 不混入无关格式化和重构

如果必须修改他人文件，先找队长确认，由队长决定是否拆单独 PR。
