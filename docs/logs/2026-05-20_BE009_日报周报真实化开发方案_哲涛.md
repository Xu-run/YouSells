## 1. 需求摘要

将 `ReportServiceImpl` 从硬编码样例数据切换为真实 MySQL 持久化，涉及两张表：`daily_reports` 和 `weekly_reports`。

## 2. 涉及数据库表

### 2.1 `daily_reports`

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| id | BIGINT | PK AUTO | 主键 |
| report_date | DATE | NOT NULL | 日报日期 |
| user_id | BIGINT | NOT NULL | 提交人ID |
| today_work | TEXT | NOT NULL | 今日完成工作 |
| issues | TEXT | NULL | 遇到的问题 |
| tomorrow_plan | TEXT | NOT NULL | 明日计划 |
| created_at | DATETIME | NOT NULL | 创建时间 |
| updated_at | DATETIME | NOT NULL | 更新时间 |
| is_deleted | TINYINT(1) | NOT NULL DEFAULT 0 | 逻辑删除 |
| **UNIQUE** | (report_date, user_id) | — | 同一人同一天只能一条日报 |

### 2.2 `weekly_reports`

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| id | BIGINT | PK AUTO | 主键 |
| week_key | VARCHAR(32) | NOT NULL | 周标识，如 `2026-W21` |
| user_id | BIGINT | NOT NULL | 提交人ID |
| weekly_summary | TEXT | NOT NULL | 本周总结 |
| issues | TEXT | NULL | 遇到的问题 |
| next_week_plan | TEXT | NOT NULL | 下周计划 |
| created_at | DATETIME | NOT NULL | 创建时间 |
| updated_at | DATETIME | NOT NULL | 更新时间 |
| is_deleted | TINYINT(1) | NOT NULL DEFAULT 0 | 逻辑删除 |
| **UNIQUE** | (week_key, user_id) | — | 同一人同一周只能一条周报 |

### 2.3 与 task_boards 表的关键差异

> ⚠️ 这两张表**没有** `created_by` / `updated_by` 字段，与 `task_boards` 不同。Entity 不能直接继承 `BaseEntity`（BaseEntity 包含这两个字段的自动填充，映射到不存在的列会导致 MyBatis-Plus 报错）。

## 3. 现有架构现状

### Controller（无需修改）
- `GET  /api/reports/daily?page=1&pageSize=20` → `reportService.pageDailyReports(page, pageSize)`
- `POST /api/reports/daily` → `reportService.createDailyReport(request)`
- `GET  /api/reports/weekly?page=1&pageSize=20` → `reportService.pageWeeklyReports(page, pageSize)`
- `POST /api/reports/weekly` → `reportService.createWeeklyReport(request)`

### Service 接口（无需修改）
- `pageDailyReports(int page, int pageSize)` → `PageResponse<DailyReportVo>`
- `createDailyReport(DailyReportCreateRequest)` → `Long`
- `pageWeeklyReports(int page, int pageSize)` → `PageResponse<WeeklyReportVo>`
- `createWeeklyReport(WeeklyReportCreateRequest)` → `Long`

### 当前实现问题
- 所有方法返回硬编码样例数据（2 条日报 + 2 条周报）
- `createDailyReport` / `createWeeklyReport` 只返回假 ID（7201L / 7301L），不做持久化
- 无重复提交检查

### 6.4 ServiceImpl 改造

| 方法 | 改造逻辑 |
|------|----------|
| `pageDailyReports` | 调用 `dailyReportMapper.selectPageWithUser(Page.of(page, pageSize))` → 返回 PageResponse |
| `createDailyReport` | 从 `SecurityUserContext.requireCurrentUser()` 获取 userId → `ReportConvert.toDailyEntity(request, userId)` → `insert(entity)` → 捕获唯一键冲突抛 `STATUS_CONFLICT` → 返回 `entity.getId()` |
| `pageWeeklyReports` | 调用 `weeklyReportMapper.selectPageWithUser(Page.of(page, pageSize))` → 返回 PageResponse |
| `createWeeklyReport` | 同上流程，冲突抛 `STATUS_CONFLICT` |

**重复提交处理策略**：
- 数据库已有 UNIQUE 约束，插入时如冲突会抛 `DuplicateKeyException`
- Service 层捕获后转为 `BusinessException(ErrorCodeConstants.STATUS_CONFLICT, "该日期/周的日报/周报已提交")`
- 注意：`ErrorCodeConstants.STATUS_CONFLICT` 当前值为 `4009`，但 `GlobalExceptionHandler` 中对应的是 `4090`（不一致）。不改 GlobalExceptionHandler（不是我的模块），使用 `4009` 即可，异常处理会走默认 BAD_REQUEST 分支返回 400。
