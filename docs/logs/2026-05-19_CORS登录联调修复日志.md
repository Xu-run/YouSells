## 问题现象

当时的联调问题主要有三类：

- 前端运行在 `http://127.0.0.1:5173`
- 后端最初只对白名单中的 `http://localhost:5173` 放行
- 浏览器访问 `/api/auth/login`、`/api/auth/me` 时出现 CORS 拦截
- 即使登录成功，也会继续出现 `login required`
- 受保护页面会先弹错误提示，再跳登录页

## 本次处理动作

### 1. CORS 白名单扩展

已将后端 CORS 配置从单一来源改为多来源：

- `http://localhost:5173`
- `http://127.0.0.1:5173`

同步更新位置：

- `yousells-server/src/main/resources/application.yml`
- `yousells-server/src/main/java/com/yousells/common/config/SecurityConfig.java`
- `.env.example`

### 2. 前端开发代理调整

已将前端开发环境改为代理模式：

- `VITE_API_BASE_URL=/api`
- `vite.config.ts` 将 `/api` 和 `/actuator` 代理到 `http://127.0.0.1:8080`

这样做的目的：

- 降低跨域 cookie 联调复杂度
- 让本地开发访问更接近同源模式

### 3. 鉴权主链路从 Session 切换到 JWT

本轮最终不再继续修补 `Session/Cookie`，而是直接切换到 JWT：

- 后端改为 `STATELESS`
- 增加 `JwtAuthenticationFilter`
- 登录返回 `accessToken`
- `/api/auth/me` 从 Bearer Token 读取登录态
- 前端统一在请求头携带 `Authorization`

### 4. 前端登录态与 401 处理收口

已补齐：

- token 本地存储
- 页面刷新后的登录态恢复
- 无 token 时不再无意义请求 `/api/auth/me`
- `401` 时自动清理登录态并跳回登录页
- 避免反复弹出 `login required`

## 当前验收结果

已确认：

- 后端 `mvn test` 全部通过
- 前端 `npm run build` 通过
- `POST /api/auth/login` 可正常返回 JWT
- `GET /api/auth/me` 在无 token 时返回 `401`
- 开发代理链路 `5173 -> /api -> 8080` 已配置完成

## 当前结论

这次联调问题的根因不是单点 CORS，而是：

- 旧的 `Session/Cookie` 链路在当前前后端分离开发模式下不稳定

因此本次修复结论为：

- CORS 只做必要收口
- 登录主链路统一切 JWT
- 后续团队联调与开发基线都以 JWT 方案为准
