# 鉴权与权限模型

## 1. 当前方案

当前 `YouSells P0` 已切换为：

- 后端：`Spring Security + JWT Bearer Token`
- 前端：`localStorage` 保存 `accessToken`
- 请求方式：统一通过 `Authorization: Bearer <token>` 访问受保护接口

本轮不再使用 `Session/Cookie` 作为前后端联调主链路。

## 2. 这样做的原因

当前项目更需要的是：

- 本地联调稳定
- 避免 `CORS + cookie + SameSite` 反复踩坑
- 避免登录后仍然出现 `login required`
- 让 `127.0.0.1:5173 -> /api -> 8080` 的开发链路更简单
- 为后续可能的微服务拆分预留空间

## 3. 当前 JWT 设计

### 3.1 Token 类型

当前只实现：

- `access token`

当前暂不实现：

- `refresh token`
- Redis 黑名单
- 强制失效旧 token
- 多端登录控制

### 3.2 Token 载荷

当前 token 中包含：

- `userId`
- `username`
- `displayName`
- `roles`

### 3.3 有效期

当前默认有效期：

- `43200` 秒
- 即 `12` 小时

配置位置：

- `yousells-server/src/main/resources/application.yml`
- 配置项：`app.jwt.access-token-expire-seconds`

### 3.4 前端存储位置

当前前端将 token 保存在：

- `localStorage`

键名：

- `yousells.access_token`

## 4. 登录态流转

### 4.1 登录

前端调用：

- `POST /api/auth/login`

后端返回：

- `accessToken`
- `tokenType`
- `expiresIn`
- `userInfo`

前端收到后：

- 保存 `accessToken`
- 保存当前用户信息
- 跳转到受保护页面

### 4.2 刷新页面

页面刷新后：

- 前端先检查本地是否存在 token
- 有 token 才请求 `GET /api/auth/me`
- 无 token 不再无意义请求 `/api/auth/me`

### 4.3 失效处理

当后端返回 `401` 时：

- 前端清空本地 token
- 清空当前用户状态
- 跳转回 `/login`

当前已经额外处理：

- 避免在失效跳转时反复弹出 `login required`

## 5. 当前角色模型

P0 先保留最小角色集：

- `ADMIN`
- `MEMBER`

演示账号：

- `admin / admin123`
- `member / member123`

## 6. 权限边界

### 所有登录成员可访问

- 首页看板
- 客户列表
- 客户详情
- 跟进记录
- 公共安排
- 日报 / 周报
- 话术库

### 后续需要继续细化的权限

后续正式接真实用户表后，建议补齐：

- 客户编辑权限
- 负责人变更权限
- 报表全量查看权限
- 话术库编辑权限
- 管理员专属配置权限

## 7. 代码边界约束

为了后续继续扩展，当前约束为：

- 业务层不直接依赖 `HttpSession`
- 当前登录人统一从 `SecurityContext` 读取
- `/api/auth/*` 统一承接登录态相关接口
- 前端不再依赖 cookie 登录态

## 8. 后续升级方向

如果后面项目要正式长期上线，建议按顺序升级：

1. 增加 `refresh token`
2. 增加 Redis token 黑名单
3. 接入真实用户表、角色表、权限点表
4. 增加细粒度菜单权限和按钮权限
5. 如需拆分服务，再将 JWT 扩展为统一认证基线
